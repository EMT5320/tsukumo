# C1 Cross-Runtime Evidence — Technical Design

## Scope

This child adds the Codex runtime profile/decoder and produces the controlled
cross-runtime evidence required before UI work. It reuses host lifecycle,
checkpoint selection, and production receipts. It does not add presentation or
release infrastructure.

## Codex Runtime Profile

Reference command shape:

```text
stdin -> codex exec --json --ephemeral --sandbox <least-capable-profile> -
```

The profile records `codex --version`, never reads/persists `auth.json`, and may
use a controlled ignore-user-config option only when repository instructions
remain explicitly part of the case contract.

The decoder maps documented JSONL classes into shared payload semantics:

- thread and turn lifecycle;
- command/tool item start, update, and completion;
- file changes, MCP calls, web searches, plans, and messages through the
  smallest honest normalized event or an observable skip;
- malformed known events and terminal errors into typed failures.

Vendor IDs remain namespaced provenance/correlation hints. Host owns durable
EventIds, execution/projection attribution, Chronicle ordering, and theater
fan-out.

## Deterministic Comparison Bundle

The V0 comparison is a bounded evidence structure generated by tests or an
explicit evidence command:

```rust
pub struct ComparisonManifest {
    pub schema_version: u16,
    pub case_id: CaseId,
    pub source_runtime: RuntimeVersion,
    pub target_runtime: RuntimeVersion,
    pub repository_fixture: ContentDigest,
    pub target_state: StateId,
    pub with_state: ComparisonRun,
    pub without_state: ComparisonRun,
    pub invariants: Vec<InvariantCheck>,
    pub summary: ComparisonSummary,
}
```

The two runs share repository fixture, goal, runtime/model configuration,
sandbox, checkpoint, renderer version, and all non-target state. The only input
change is target StateId inclusion plus the digests derived from rendered
bytes. The manifest records selected refs, production receipt IDs/digests,
normalized tool arguments/outcomes, and invariant results.

Raw prompts, credentials, home paths, and uncontrolled repository contents are
rejected. Test outputs use temporary directories; a reviewed redacted fixture
may be committed. No general snapshot table, expiry scheduler, retain API, or
cleanup audit is introduced in V0.

## Claim Boundary

- A normal run may state that the GNU StateRef was selected/projected and a GNU
  tool call followed.
- The paired run may state that removing the state changed observed tool
  arguments under the recorded controls.
- The result does not generalize across every model, task, or configuration.

## Post-Revoke Flow

Host applies a user-evidenced revoke transition, compiles the next checkpoint,
and prepares a new projection. The new receipt excludes the StateRef; the old
receipt and Chronicle chain remain immutable and queryable.

## Tests

- Codex decoder event/error/truncation table tests.
- Claude/Codex normalized conformance fixtures.
- Full source event -> state -> checkpoint -> receipt -> Codex tool -> outcome
  trace.
- With/without invariant and normalized argument comparison.
- Post-revoke selection with historical receipt inspection.
- Fixture secret/path validation and receipt prompt-sentinel regression.
- Opt-in live dual-runtime smoke using the same host/decoder code.

## Rollback

Codex support and fixtures are additive. A failed live run remains diagnostic
evidence and cannot replace deterministic fixtures automatically.
